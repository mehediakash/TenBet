# Agent Hierarchy and Permission System Documentation

## Overview

This platform implements a strict hierarchy-based access control system where agents can **ONLY** see and manage data from users and agents in their downline. This ensures proper data isolation and prevents agents from accessing data outside their hierarchy.

## Hierarchy Structure

```
SUPER ADMIN (Admin)
    ↓
MASTER AGENT
    ↓
AGENT (Sub Agent)
    ↓
AGENT
    ↓
PLAYERS (Regular Users)
```

### Role Definitions

1. **SUPER_ADMIN (Admin)**
   - Has complete access to ALL data across the platform
   - Can create Master Agents
   - Bypasses all permission checks
   - Sees all users, agents, transactions, and reports

2. **MASTER_AGENT**
   - Can create Agents and Sub Agents
   - Sees ONLY users/agents created under them or their downline
   - Receives commissions from their entire downline
   - Can manage their downline's settings (if permissions granted)

3. **AGENT / SUB_AGENT**
   - Can create players and possibly sub-agents (if permission granted)
   - Sees ONLY users/agents they created or in their downline
   - Cannot see data from other agents' downlines
   - Receives commissions from their downline only

4. **USER (Player)**
   - Regular platform users
   - No downline management capabilities
   - Can only see their own data

## Data Access Rules

### ✅ What Agents CAN Access

- **Users**: Only users where the agent is in their hierarchy chain
- **Transactions**: Only transactions from their downline users
- **Commission**: Only commissions generated by their downline
- **Reports**: Only data scoped to their downline
- **Dashboard Stats**: Aggregated from their downline only

### ❌ What Agents CANNOT Access

- Users created by other agents outside their hierarchy
- Transactions from users not in their downline
- Other agents' commission data
- Platform-wide statistics
- Admin-only features

## Server-Side Filtering

### Downline Query Builder

The server uses a dynamic query builder to filter data by hierarchy:

```javascript
getDownlineQuery(agentId, agentRole) {
  switch (agentRole) {
    case 'master_agent':
      return {
        $or: [
          { 'hierarchy.masterAgent': agentId },
          { 'hierarchy.agent': agentId },
          { 'hierarchy.subAgent': agentId }
        ]
      };
    case 'agent':
      return {
        $or: [
          { 'hierarchy.agent': agentId },
          { 'hierarchy.subAgent': agentId }
        ]
      };
    case 'sub_agent':
      return {
        'hierarchy.subAgent': agentId
      };
    default:
      return { _id: agentId }; // Only self
  }
}
```

### Usage in Controllers

```javascript
// In agentDashboardController.js
async getUserStatistics(agentId, agentRole) {
  const query = this.getDownlineQuery(agentId, agentRole);
  const totalUsers = await User.countDocuments(query);
  // ... more stats
}

// In transactionApprovalController.js
async getPendingTransactions(req, res) {
  const downlineUsers = await User.find({
    $or: [
      { 'hierarchy.masterAgent': req.user.id },
      { 'hierarchy.agent': req.user.id },
      { 'hierarchy.subAgent': req.user.id }
    ]
  }).select('_id');

  const downlineIds = downlineUsers.map(user => user._id);

  transactions = await Deposit.find({
    user: { $in: downlineIds },
    status: 'pending'
  });
}
```

## Frontend Implementation

### usePermissions Hook

The `usePermissions` hook provides centralized permission checking:

```javascript
const { can, isAdmin, role } = usePermissions();

// Usage in components
if (can.viewUsers) {
  // Show users list
}

if (can.createSubAgents) {
  // Show create agent button
}

// Admin automatically has all permissions
// isAdmin || hasPermission(...) pattern
```

### Conditional API Calls

Components check user role to call appropriate endpoints:

```javascript
// In UserManagement.jsx
const loadUsers = async () => {
  const response = isAdmin
    ? await userAPI.getUsers(params) // All users
    : await userAPI.getDownlineUsers(params); // Only downline
};

// In AgentManagement.jsx
const loadAgents = async () => {
  const response = isAdmin
    ? await agentAPI.getAgents() // All agents
    : await agentAPI.getDownlineAgents(); // Only sub-agents
};

// In TransactionManagement.jsx
const loadPendingTransactions = async () => {
  const response = isAdmin
    ? await paymentAPI.getPendingTransactions({ type })
    : await paymentAPI.getDownlinePendingTransactions({ type });
};
```

## Commission Flow

Commissions flow upward through the hierarchy:

```
Player bets $100
    ↓
Sub Agent gets 2% = $2
    ↓
Agent gets 3% = $3
    ↓
Master Agent gets 5% = $5
```

Each level's commission is calculated based on:

- Loss Commission: Percentage of player losses
- Turnover Commission: Percentage of total bet volume
- Profit Share: Percentage of downline profits

## Testing Hierarchy Access

### Master Agent Test Scenario

1. **Login as Master Agent**
   - Should see dashboard stats from downline only
   - User count = users created by them + their agents + sub-agents
   - Should NOT see users created by other master agents

2. **Navigate to User Management**
   - Should only see users in hierarchy chain
   - Cannot see players created by other agents

3. **Navigate to Transaction Management**
   - Should only see transactions from downline users
   - Cannot approve transactions from other hierarchies

4. **Navigate to Agent Management**
   - Should only see agents they created
   - Cannot see other master agents' sub-agents

### Agent Test Scenario

1. **Login as Agent**
   - Dashboard shows only their downline stats
   - User count = users they created + their sub-agents' users

2. **Create Sub-Agent** (if permission granted)
   - New sub-agent has `hierarchy.agent = agentId`
   - Sub-agent can only see users they create

3. **View Reports**
   - All reports scoped to downline only
   - Commission reports show only their earnings

## Security Considerations

1. **Server-Side Validation**
   - NEVER trust client-side role/permission checks
   - Always validate hierarchy in backend
   - Use middleware to enforce authorization

2. **Database Queries**
   - Always include hierarchy filter in queries
   - Use `getDownlineQuery()` helper consistently
   - Validate agent can access requested resource

3. **API Route Protection**

   ```javascript
   // Protect routes with role-based middleware
   router.use(protect); // Verify JWT
   router.use(authorize("master_agent", "agent", "sub_agent")); // Check role
   ```

4. **Permission Checks**
   ```javascript
   // In controller, verify permission from AgentSettings
   const settings = await AgentSettings.findOne({ agent: req.user.id });
   if (!settings.permissions.approveDeposits) {
     return res.status(403).json({ message: "Permission denied" });
   }
   ```

## Common Issues & Solutions

### Issue: Master Agent sees all platform users

**Cause**: Frontend calling admin endpoint instead of agent endpoint

**Solution**:

```javascript
// Wrong
const response = await userAPI.getUsers();

// Correct
const response = isAdmin
  ? await userAPI.getUsers()
  : await userAPI.getDownlineUsers();
```

### Issue: Permission checks not working

**Cause**: Using old `hasPermission` function instead of `usePermissions` hook

**Solution**:

```javascript
// Wrong
const canEdit = hasPermission(user, PERMISSIONS.EDIT_USERS);

// Correct
const { can } = usePermissions();
const canEdit = can.editUsers;
```

### Issue: Transactions not filtered by hierarchy

**Cause**: API endpoint doesn't filter by downline

**Solution**: Use `/api/agent-transactions/pending` instead of `/api/admin/transactions/pending`

## Best Practices

1. **Always use `usePermissions` hook** instead of direct permission checks
2. **Check `isAdmin` first** in conditional logic for optimal performance
3. **Use hierarchy-scoped endpoints** for agent roles
4. **Validate permissions server-side** even if UI hides buttons
5. **Test with different roles** to ensure proper data isolation
6. **Log API responses** during development to verify correct data filtering
7. **Document permission requirements** for new features
8. **Update both frontend and backend** when adding new permissions
